# Part A — M1 抽取引擎

## A1. M1 目标与在路线图中的位置

M0 已跑通骨架（喂 jar → 两类白送元数据 → 分片 → manifest → 会话库 → `find-config-properties` → CLAUDE.md，Kotlin 抽取器 + Go 查询器 + SQLite 契约）。

**M1 是项目的护城河：用 ASM 读字节码，把 Spring 的隐式事实抽全。** M1 完成后，分片里不再只有配置项，而是有类层次、接口实现、注解、`@Bean`、条件装配、SPI/扩展点、字符串常量等。

M1 的验证方式以**黄金 jar 单元测试**（断言抽取器对已知 jar 产出预期事实）为主，并额外实现**两个查询命令**专门验证本项目的中心设计——「存符号引用、跨 jar 关系留到查询期 join」是否端到端成立。完整命令集与会话库缓存留给 M2。

## A2. 锁定决策（不要重新评估）

- **抽取器**：Kotlin/JVM。M1 **引入 ASM（`org.ow2.asm:asm`，用较新稳定版）**。
- **⚠️ 重要修正：M1 不依赖 spring-core，纯 ASM 抽取。** 原因见下框。
- **查询器**：Go，`modernc.org/sqlite`（纯 Go、零 CGO），对 SQLite 只读。
- **契约**：SQLite 文件。两端只通过它通信。
- **分片身份**：`shard_id = <jar_sha256>@<extractor_version>`。

> **为什么放弃 spring-core / 元注解展开移到查询期（务必理解）**
> spring-core 的 `MetadataReader` 会做元注解展开（`@RestController`→`@Controller`+`@ResponseBody`）、`@AliasFor` 合并。但元注解展开**本质是跨 jar 的**（`@Service` 定义在 spring-context，使用它的类在业务 jar）。在「每个 jar 独立抽取、分片可跨项目共享/可预生成」的模型下：
> - 若给它全 classpath 的 ResourceLoader → 抽取一个 jar 依赖整体上下文，**破坏分片隔离性**（无法单独缓存/分发/预生成）。
> - 若只给当前 jar → 跨 jar 元注解无法解析，**展开不全且不确定**（同一 jar 在不同环境产出不同分片），破坏可复现与可共享。
>
> 因此 M1 改为：**纯 ASM 只抽「直接出现」的注解（存为符号引用 FQN，不展开）**。注解类本身也是普通类——当它所在 jar 被索引时，它的 `classes` 行（kind=`annotation`）与其上的 `@Component` 等注解都会被记录。于是**元注解关系作为事实被自然存下**，展开变成查询期对合并后注解图的遍历（M2）。这与项目「存符号引用、关系留到查询期」的中心原则完全一致，且让抽取器零 Spring 依赖、纯确定性。

## A3. M1 范围

### A3.1 做什么（IN）

1. **抽取器扩展**：对每个 jar 的每个 `.class` 条目跑 ASM，抽出类/方法/注解/继承/接口/`@Bean`/条件/字符串常量；并把 SPI/扩展点抽全。具体到表的规则见 A5、A6。
2. **版本号提升**：`schema_version = 2`、`extractor_version = 2`（见 A4），M0 分片自动被取代、自动重建。
3. **会话库合并扩展**：合并新表中 M1 命令需要的那几张（见 A7）。
4. **两个验证用查询命令**：`find-implementations` 与 `find-by-annotation`（见 A8）。
5. **黄金 jar 测试 + actuator 真值校验**（见 A11）。

### A3.2 明确不做（OUT）

- ❌ **元注解展开 / `@AliasFor` 解析**（M2 查询期做）。M1 只存直接注解。
- ❌ **FTS5 全文检索表**（`config_properties_fts` / `string_constants_fts`）。M1 仅存 `string_constants` 普通表，不建 FTS。（FTS 与 `find-string` 一起放 M2；这样也回避 Go 端 `modernc.org/sqlite` 是否支持 FTS5 的开库风险。）
- ❌ **字段级注解、`fields` 表**。M1 只把「常量字符串字段」抽进 `string_constants`，不抽字段注解。
- ❌ **`get-class` / `explain-conditional` / `find-string` / `get-bean-definitions` / `list-extension-points` 等其余命令**（M2）。
- ❌ **`find-implementations` 的传递闭包**（子接口/父类链）。M1 只做**直接声明**匹配（见 A8）。
- ❌ **会话库按 fingerprint 全局去重/缓存复用、LRU/GC、方法级调用图、mvn/gradle 解析、MCP、多宿主探测、分片签名/下载**——分别属于 M2/M3/M5，不在 M1。
- ❌ **抽取器版本迁移脚本**：无需（见 A4）。

## A4. 版本号变更与「自动失效」

- `schema_version`：`1 → 2`（新增多张分片表）。
- `extractor_version`：`1 → 2`（抽取产出大幅增加）。
- **效果即设计**：因 `shard_id` 含 `extractor_version`，所有 M0 的 `…@1` 分片自动被 `…@2` 取代；M1 首次运行会对全部 jar 重新抽取一次，**无需手动清缓存、无需迁移脚本**。
- 抽取器索引时始终以**当前** `extractor_version`(=2) 为目标 `shard_id`；命中则跳过，否则抽取。旧 `@1` 分片留在磁盘、被忽略（清理属 M5 的 GC）。
- manifest 的 `shards` 表结构 M0→M1 不变，无 manifest 迁移。会话库是临时重建产物，无迁移。

## A5. 数据模型：M1 新增的分片表 DDL（完整）

M1 在 M0 已有的 `shard_meta`、`spi_registrations`、`config_properties` 基础上**新增**下列表。命名须与本文严格一致（完整 schema 的前向兼容子集）。

```sql
CREATE TABLE classes (
  id          INTEGER PRIMARY KEY,
  fqn         TEXT NOT NULL,        -- 点号全限定名
  kind        TEXT,                 -- class|interface|enum|annotation|record
  super_fqn   TEXT,                 -- 符号引用，可空（接口/Object 为空）
  modifiers   INTEGER,              -- ASM access flags
  is_abstract INTEGER,
  source_file TEXT                  -- SourceFile 属性，可空
);
CREATE INDEX idx_classes_fqn   ON classes(fqn);
CREATE INDEX idx_classes_super ON classes(super_fqn);

CREATE TABLE class_interfaces (    -- 直接声明实现/继承的接口，符号引用
  class_id      INTEGER NOT NULL,
  interface_fqn TEXT NOT NULL
);
CREATE INDEX idx_ci_iface ON class_interfaces(interface_fqn);
CREATE INDEX idx_ci_class ON class_interfaces(class_id);

CREATE TABLE methods (
  id         INTEGER PRIMARY KEY,
  class_id   INTEGER NOT NULL,
  name       TEXT NOT NULL,
  descriptor TEXT NOT NULL,         -- JVM 方法描述符（区分重载）
  return_fqn TEXT,                  -- 返回类型点号名
  modifiers  INTEGER
);
CREATE INDEX idx_methods_class ON methods(class_id);

CREATE TABLE annotations (          -- 仅「直接出现」的注解，不展开元注解
  id             INTEGER PRIMARY KEY,
  target_kind    TEXT NOT NULL,     -- M1: 'class' | 'method'
  target_id      INTEGER NOT NULL,  -- class_id 或 method_id
  annotation_fqn TEXT NOT NULL,     -- 符号引用
  attributes     TEXT               -- JSON：属性名 -> 值（数组/嵌套转 JSON 文本）
);
CREATE INDEX idx_ann_fqn    ON annotations(annotation_fqn);
CREATE INDEX idx_ann_target ON annotations(target_kind, target_id);

CREATE TABLE bean_definitions (     -- @Bean 方法
  id            INTEGER PRIMARY KEY,
  config_fqn    TEXT NOT NULL,      -- 声明该 @Bean 方法的类 FQN
  method_id     INTEGER,
  bean_type_fqn TEXT,               -- @Bean 方法返回类型（符号引用）
  bean_name     TEXT
);
CREATE INDEX idx_bean_type ON bean_definitions(bean_type_fqn);
CREATE INDEX idx_bean_cfg  ON bean_definitions(config_fqn);

CREATE TABLE conditions (           -- 结构化 @ConditionalOn* / @Conditional
  id          INTEGER PRIMARY KEY,
  target_kind TEXT NOT NULL,        -- 'class' | 'bean_method'
  target_id   INTEGER NOT NULL,
  type        TEXT NOT NULL,        -- OnClass|OnMissingClass|OnBean|OnMissingBean|OnProperty|OnExpression|OnWebApplication|OnResource|Conditional|...
  ref_value   TEXT                  -- JSON：与该条件相关的属性，如 {"value":["..."],"property":"x","havingValue":"true","matchIfMissing":false}
);
CREATE INDEX idx_cond_target ON conditions(target_kind, target_id);

CREATE TABLE string_constants (     -- 常量池/字段常量中的字符串
  id        INTEGER PRIMARY KEY,
  class_id  INTEGER NOT NULL,
  method_id INTEGER,                -- 来自方法体则填，来自常量字段则 NULL
  value     TEXT NOT NULL
);
CREATE INDEX idx_strconst_class ON string_constants(class_id);
```

> M1 不建 `config_properties_fts` / `string_constants_fts`（FTS 属 M2）。不建 `fields` 表。`annotations.target_kind` 在 M1 只用 `class`/`method`（`field` 留作前向兼容，不产出）。

## A6. 抽取逻辑（纯 ASM，逐项精确规则）

读 jar 用 `java.util.zip.ZipFile`；**不加载任何类、不执行任何 jar 代码**。对每个 `.class` 字节用 ASM `ClassReader` 跑一个 `ClassVisitor`。`ClassReader` 需**保留方法 code**（要读 `LDC` 字符串）与 `SourceFile` 属性；可跳过 stack-map frames 以提速。对超出 ASM 支持的 class 文件版本，**捕获异常→跳过该 class→记 warning→不使主 jar 失败**。

按一次 ASM pass 产出各表：

**`classes` / `class_interfaces`（`visit` 回调）**
- `fqn` = 由内部名转点号名。`super_fqn` = 父类点号名（`java/lang/Object` 或空 → NULL）。
- `kind`：按 access 标志判定 `interface`/`annotation`/`enum`/`record`/否则 `class`（record 可用 `ACC_RECORD` 或父类 `java.lang.Record` 判定；判不准记 `class`）。
- `modifiers` = access；`is_abstract` = 含 `ACC_ABSTRACT`。
- 每个声明的接口 → 一条 `class_interfaces`（`interface_fqn` 点号名）。
- `source_file` 来自 `visitSource`。

**类级注解（`visitAnnotation`）**
- `annotation_fqn` = 注解描述符转点号名。用 `AnnotationVisitor` 收集属性为 `map`→JSON，写 `annotations`（`target_kind='class'`, `target_id`=该类 id）。
- **条件识别**：若 `annotation_fqn` 以 `org.springframework.boot.autoconfigure.condition.ConditionalOn` 开头，或等于 `org.springframework.context.annotation.Conditional` → 另写一条 `conditions`（`target_kind='class'`, `type`=注解简单名去掉前缀 `Conditional`/保留 `OnXxx`，`ref_value`=该注解属性 JSON）。

**方法（`visitMethod`）**
- 写 `methods`（`class_id`, `name`, `descriptor`, `return_fqn`=返回类型点号名, `modifiers`）。**跳过 `ACC_SYNTHETIC` 与 `ACC_BRIDGE` 方法**（编译器噪音）。
- 方法上的 `MethodVisitor`：
  - `visitAnnotation` → 写 `annotations`（`target_kind='method'`, `target_id`=该方法 id）+ 同上的条件识别（但 `target_kind='bean_method'`, `target_id`=方法 id）。
  - `visitLdcInsn(cst)`：若 `cst` 为 `String` → 写 `string_constants`（`class_id`, `method_id`=该方法 id, `value`）。
- **`@Bean` 识别**：若方法直接注解了 `org.springframework.context.annotation.Bean` → 写 `bean_definitions`：`config_fqn`=声明类 FQN，`method_id`=该方法 id，`bean_type_fqn`=方法返回类型点号名，`bean_name`=`@Bean` 的 `name`/`value` 属性（取第一个）否则方法名。

**常量字段（`visitField`）**
- 若 `visitField` 的 `value` 为非空 `String` 常量 → 写 `string_constants`（`class_id`, `method_id`=NULL, `value`）。
- M1 **不**抽字段注解。

**SPI / 扩展点（非 `.class` 的元数据文件）→ `spi_registrations`**
- `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`：每行（去注释/空行）→ `mechanism='autoconfig.imports'`, `key=NULL`, `impl_fqn`=FQN。（同 M0）
- `META-INF/spring.factories`（properties 格式，处理 `\` 续行）：**对每一个 key**（不止 EnableAutoConfiguration），其逗号分隔的每个 FQN → `mechanism='spring.factories'`, `key`=该 factory key, `impl_fqn`=FQN。（M0 只做了一个 key，M1 做全部）
- `META-INF/services/<完整接口FQN>`：文件名即服务接口 FQN，文件内每行（去注释/空行）即实现 FQN → `mechanism='services'`, `key`=服务接口 FQN, `impl_fqn`=实现 FQN。

**配置项与坐标**：与 M0 完全一致（`config_properties` 来自 `spring-configuration-metadata.json`/`additional-…`；`jar_coordinate` 来自 `META-INF/maven/**/pom.properties`，缺失则 NULL）。

## A7. 噪音过滤与性能

- 方法表跳过 `ACC_SYNTHETIC`/`ACC_BRIDGE`；字符串常量仅取这些非合成方法体内的（及常量字段）。
- 保留内部类/嵌套类的 `classes` 行（它们可能承载注解/接口实现）。
- **每个 jar 的写入放在单个 SQLite 事务**里，行批量插入；**跨 jar 并行抽取**（线程池，每线程独立 SQLite 连接/独立分片文件）。
- 分片仍**原子落盘**（temp → fsync → rename）；manifest 仍 `INSERT OR IGNORE` + WAL + `busy_timeout`。

## A8. 会话库合并扩展（M1）

M1 会话库在 M0 已合并的 `config_properties`、`spi_registrations` 之外，**新增合并** `classes`、`class_interfaces`、`annotations` 三张（M1 两个命令所需）。`methods`/`bean_definitions`/`conditions`/`string_constants` 已写入分片但**暂不合并**（等 M2 对应命令再合并，保持 M1 会话库精简）。

合并方式不变：**逐个分片 `ATTACH → INSERT…SELECT…,'<shard_id>' → DETACH`，任一时刻只挂载 1 个分片**，规避 ATTACH 数量上限。会话库各表加 `source_shard_id TEXT NOT NULL` 列并重建索引（含 `classes(fqn)`、`class_interfaces(interface_fqn)`、`annotations(annotation_fqn)`、`annotations(target_kind,target_id)`）。

## A9. M1 的两个验证用查询命令

二者都：定位 `.research4jar/project.json` → 只读打开会话库 → 查询 → 输出 JSON（含 `coverage`，取自 project.json，与 M0 一致）。支持 `--project-dir` / `--format json|text` / `--page` / `--page-size`。

### A9.1 `find-implementations <interface-or-class-fqn>`

验证「存符号引用、跨 jar join」的中心设计（接口在 A jar、实现散落在 B/C jar）。

```sql
SELECT c.fqn, c.source_shard_id
FROM classes c
WHERE c.id IN (SELECT class_id FROM class_interfaces WHERE interface_fqn = ?)
   OR c.super_fqn = ?
ORDER BY c.fqn;
```

- **M1 只做直接声明匹配**（类自身 `implements`/`extends` 中直接出现该 FQN）。
- **已知限制（写进帮助与 README）**：不做传递闭包——实现了目标的子接口、或继承了实现该接口的父类的类，不会被返回。传递闭包（基于 `super_fqn` + 接口图的递归遍历）属 M2。
- 典型用例：`jakarta.servlet.Filter`、`org.springframework.beans.factory.config.BeanPostProcessor`、`org.springframework.boot.env.EnvironmentPostProcessor`。

### A9.2 `find-by-annotation <annotation-fqn>`

```sql
SELECT c.fqn, c.source_shard_id, a.attributes
FROM annotations a JOIN classes c ON c.id = a.target_id
WHERE a.target_kind = 'class' AND a.annotation_fqn = ?
ORDER BY c.fqn;
```

- **M1 只匹配直接注解**，不展开元注解。
- **已知限制（务必写进帮助与 README，避免误解）**：查 `org.springframework.stereotype.Component` **不会**返回 `@Service`/`@Repository`/`@Controller` 标注的类（它们是被 `@Component` **元注解**的）。但展开所需数据 M1 已存好（`@Service` 注解类自身的 `classes` 行 + 其上的 `@Component` 注解行）；**元注解展开的查询期遍历属 M2**。
- 典型用例（直接注解即有效）：`org.springframework.boot.context.properties.ConfigurationProperties`、`org.springframework.context.annotation.Configuration`。

### A9.3 JSON 输出契约

```json
{
  "query": { "command": "find-implementations", "arg": "jakarta.servlet.Filter" },
  "results": [
    { "fqn": "com.example.web.TraceFilter", "source_jar": "com.example:web-common:2.3.0", "attributes": null }
  ],
  "total": 1, "page": 1, "page_size": 20,
  "coverage": { "jars_total": 312, "jars_indexed": 310, "jars_missing": ["com.acme:internal-foo:1.0.0"], "extractor_version": 2 }
}
```

`source_jar` 由该行 `source_shard_id` 反查 manifest 的 `jar_coordinate`，缺失则用 jar 文件名。`find-by-annotation` 的 `attributes` 为该注解属性 JSON；`find-implementations` 该字段为 `null`。

## A10. 确定性要求

延续 M0：给定相同 `(jar 字节, extractor_version)`，分片语义一致，以支撑后续分片共享/预生成。
- 各表插入前按稳定键排序（`classes` 按 `fqn`；`methods` 按 `(class_id, name, descriptor)`；`annotations` 按 `(target_kind, target_id, annotation_fqn)`；`string_constants` 按 `(class_id, value)`；`spi_registrations` 按 `(mechanism, key, impl_fqn)`）。
- `shard_meta.created_at = 0`。`attributes`/`ref_value` 的 JSON **对象键有序输出**（保证字节一致）。
- ASM 全程确定性；不引入任何随机/时间相关字段进分片。

## A11. M1 验收标准（Definition of Done）

1. **全量抽取跑通**：对一个真实 Spring Boot 微服务的依赖 jar 目录运行 `research4jar-index`，无异常完成，新表被填充，统计含 total/indexed/skipped/missing。
2. **黄金 jar 测试通过**：维护一组事实已知的知名 Spring jar（如 `spring-boot-autoconfigure`、`spring-web`、`spring-context`），单元测试断言抽取结果，至少覆盖：自动配置类数量（`spi_registrations` mechanism=`autoconfig.imports`）、某已知 `@ConfigurationProperties` 类被 `find-by-annotation` 命中、某已知 `@Bean` 进入 `bean_definitions`、某已知 `@ConditionalOnClass` 进入 `conditions` 且 `ref_value` 正确、某已知 `Filter`/`BeanPostProcessor` 实现被 `find-implementations` 命中、某已知 header/配置 key 字符串进入 `string_constants`。
3. **跨 jar join 成立**：`find-implementations jakarta.servlet.Filter` 能返回**位于其它 jar** 的实现类（证明符号引用 + 查询期 join 端到端工作）。
4. **actuator 真值校验（开发期预言机）**：跑一个真实 Spring Boot 应用，`--debug` 取 CONDITIONS EVALUATION REPORT 与 `/actuator/beans`、`/actuator/conditions`；写一个对照测试，抽样核对静态抽取的自动配置/条件/Bean 与运行时真值一致（不一致项列出供排查）。
5. **增量命中**：第二次跑同样 index，所有 `@2` 分片因内容寻址命中而 skipped，耗时显著低于首次。
6. **确定性**：同一 jar 抽两次，分片内容（除文件系统元信息外）一致。
7. **健壮**：输入含损坏 jar / 超新版本 class 时，跳过并计入 warning/`jars_missing`，不中断整体。
8. **版本自动失效**：在已有 M0 (`@1`) 分片的环境跑 M1，自动以 `@2` 重抽，旧分片不影响结果。

# Part B — GitHub 私有仓库与开源化

## B1. 总体策略

**先私有开发，但脚手架按开源标准一次铺好；待你人工确认后再翻公开。** 即：现在创建**私有**仓库并推送，同时把 LICENSE/README/CONTRIBUTING/CI 等开源必需件全部就位，这样将来翻公开是「一个开关 + 一次检查」，而非临时补课。**翻公开是【需人工】的决策，Codex 不得自动执行。**

仓库形态：**monorepo**，沿用 M0 目录（`indexer/` Kotlin + `querier/` Go），CI 同时构建两端。

## B2. 开源脚手架文件清单（Codex 创建）

在仓库根创建以下文件（已存在则**追加/合并**，不要覆盖用户已有内容）：

- **`LICENSE`** — 默认 **Apache-2.0**（开发工具常用，含专利授权，适合企业贡献者）。【需人工】确认许可证选择；若用户选 MIT 则换 MIT。年份与版权人留待用户填。
- **`README.md`** — 公开门面，包含：一句话定位、它解决什么问题（大型 Spring Boot 项目里 jar 级隐式依赖被 AI 漏掉）、架构概览（洋葱四层 + 一 jar 一分片 + 查询期 join）、安装与快速开始（构建 indexer 与 querier、索引一个项目、跑 `find-config-properties`/`find-implementations`/`find-by-annotation`）、能力与**已知限制**（直接注解匹配、无传递闭包——见 A9）、`coverage` 字段含义、路线图（链接 M0→M5）、项目状态标注为早期/活跃开发中、License 徽章。
- **`CONTRIBUTING.md`** — 如何构建（JDK 版本 + Gradle；Go 版本 + `go build`）、如何跑测试（黄金 jar 测试、actuator 对照测试）、代码风格、分支与 PR 流程、提交信息规范（建议 Conventional Commits）、贡献者需遵守 CoC、签署方式（建议 DCO：commit 加 `Signed-off-by`）。
- **`CODE_OF_CONDUCT.md`** — 采用 Contributor Covenant 标准文本，联系邮箱留待用户填。
- **`SECURITY.md`** — 漏洞上报方式（私下邮箱）、说明本工具**不执行 jar 代码（纯 ASM 读字节）**因此供应链面很小、下载分片（未来）会校验本地 jar hash。
- **`CHANGELOG.md`** — Keep a Changelog 风格；初始记录 M0、M1 已完成的能力。
- **`.gitignore`** — 覆盖：Kotlin/Gradle（`/indexer/build/`、`.gradle/`、`*.class`）、Go（`/querier/bin/`、可执行产物）、SQLite 本地数据（`*.db`、`.research4jar/`、任何 `RESEARCH4JAR_HOME` 默认目录的产物——但这些一般在仓库外）、IDE（`.idea/`、`.vscode/`、`*.iml`）、OS（`.DS_Store`）。**务必先于首次 commit 落地**，防止把本地分片/manifest/构建产物提交进去。
- **`.github/workflows/ci.yml`** — 见 B3。
- **`.github/ISSUE_TEMPLATE/bug_report.md` 与 `feature_request.md`**、**`.github/PULL_REQUEST_TEMPLATE.md`** — 标准模板（可选但建议）。
- **`docs/`** — 把现有设计文档（架构方案、M0、M1 说明书）放入。【需人工】用户提供这些 md 文件；Codex 不要凭空生成它们的内容。

## B3. CI 工作流要点（`.github/workflows/ci.yml`）

- 触发：`push` 到 `main` 与 `pull_request`。
- **两个 job（或一个多步 job）**：
  - **indexer**：`actions/setup-java`（选定 JDK，如 21）→ Gradle 构建 + 运行单元测试（含黄金 jar 测试）。黄金 jar 通过 Gradle 依赖解析获取或缓存，**不要把大 jar 提交进仓库**。
  - **querier**：`actions/setup-go`（选定 Go 版本）→ `go build ./...` + `go test ./...` + `go vet`。
- **跨 OS 矩阵**：至少 `ubuntu-latest` 与 `windows-latest`（Windows 是明确关注点，querier 用纯 Go 应天然通过）；`macos-latest` 可选。
- 缓存 Gradle 与 Go module 以加速。
- 失败即红，作为 PR 合并门槛。
- （发布流水线——goreleaser 跨平台构建 querier、发布 GitHub Releases——属后续，不在本次范围；可在 README 路线图标注 planned。）

## B4. git 初始化与创建私有仓库（执行步骤）

> 前提【需人工】：用户已安装并登录 GitHub CLI（`gh auth login`）或配置好 SSH/token。**Codex 不能代替用户认证 GitHub。** 仓库名与 owner 也【需人工】确认（如 `research4jar`）。

1. 确认 `.gitignore` 已就位（B2）。
2. `git init`（如尚未）；将默认分支设为 `main`：`git branch -M main`。
3. `git add -A` 后检查 `git status`，**确认没有把本地分片/manifest/构建产物加入**（应被 `.gitignore` 排除）。
4. 首次提交：`git commit -m "chore: initial commit (M0+M1, scaffolding)"`（建议带 `Signed-off-by` 以符合 DCO）。
5. 创建**私有**远端并推送，二选一：
   - **gh CLI（推荐）**：`gh repo create <owner>/<name> --private --source=. --remote=origin --push`
   - **手动**：在 GitHub 网页新建私有空仓库 → `git remote add origin <url>` → `git push -u origin main`
6. 推送后在仓库 Settings 确认：可见性=**Private**；分支保护（对 `main` 要求 PR + CI 通过，可选但建议）；Actions 已启用以触发 CI。

## B5. 「转公开」前检查清单【需人工决策】

**此步骤 Codex 不得自动执行。** 当用户决定开源时，逐项确认：

1. 仓库历史中**无任何密钥/令牌/内部凭据**（必要时用工具扫历史；如已混入，重写历史或重建仓库）。
2. **无任何专有/内部 jar、客户数据、内部包名敏感信息**被提交（分片与 manifest 不应入库；确认 `.gitignore` 一直生效）。
3. `LICENSE` 已选定并填妥版权人/年份；`README` 公开质量达标；`CODE_OF_CONDUCT`/`SECURITY` 联系邮箱已填。
4. CI 当前为绿；`CHANGELOG` 反映现状。
5. 在 GitHub Settings 将可见性切到 **Public**；如需，创建首个 `v0.x` Release 并附构建说明。
6. （可选）添加 topics、`good first issue` 标签与若干引导性 issue、把路线图（M2→M5）登记为 issues/milestones，便于外部贡献者切入。

## B6. 可持续开源运营（轻量建议，非强制）

- 语义化版本 + 持续维护 `CHANGELOG`；每个里程碑出一个 Release。
- README 顶部清晰标注「早期/活跃开发中」与支持范围，管理外部预期。
- 保持 CI 绿色作为硬门槛；PR 模板要求关联 issue 与测试。
- 把路线图（M0→M5）作为 milestones 公开，新人一眼看懂可贡献点。
- issue/PR 有基本响应节奏；用 `good first issue`/`help wanted` 降低参与门槛。

# Part C — 给 Codex 的执行顺序与总验收

**建议顺序：**
1. 先做 **Part A**：扩 schema → ASM 抽取（先 `classes`/`class_interfaces`/`annotations` 跑通 `find-implementations`/`find-by-annotation`，验证跨 jar join）→ 再补 `methods`/`bean_definitions`/`conditions`/`string_constants` 与 SPI 全量 → 黄金 jar 测试 → actuator 对照 → 增量/确定性/健壮性。逐项过 A11。
2. A11 全绿后，再做 **Part B**：铺脚手架 → CI → 初始化 git → 创建**私有**仓库并推送（认证由用户完成）。**到此为止**——「转公开」(B5) 留给用户人工决策，Codex 不执行。

**总验收 = A11 八项全部通过 + Part B 私有仓库已建立且 CI 在远端跑绿 + 仓库保持私有（未自动公开）。**

> 实现原则不变：先打通最小闭环再加厚（先 3 张表 + 2 个命令验证 join，再补全），先验证主链路再扩展。遇到「明确不做」清单内的诱惑（元注解展开、FTS、传递闭包、调用图、自动翻公开）一律按本文推迟或交人工。
