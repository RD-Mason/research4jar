# Research4Jar · M0「走通骨架」实现说明书

> **给实现者（Codex）的话**：本文件是自包含的实现规格。你无需任何对话背景，照本文逐条实现即可。请严格遵守第 3 节的「明确不做」清单——不要超范围发挥。所有命名（表名/列名/字段名）都要与本文一致，因为后续里程碑会在此基础上**增量扩展**，命名不一致会造成返工。

-----

## 1. 背景与 M0 目标

Research4Jar 是一个把 Spring Boot 微服务的 jar 依赖抽取成可查询事实库的本地工具，让 AI 编码工具精确查询 jar 里的自动配置、配置项、Bean 关系等隐式依赖，做到「不漏依赖、不烧 token」。完整形态是**跨项目全局库**：一个 jar 一个不可变分片（shard），查询时只挂载当前项目 classpath 涉及的那几百个分片。

**M0 是「走通骨架」（walking skeleton）**：一条很薄但**结构真实**的链路，目的是端到端验证这条主链路——

```
喂入 jar → 抽取「白送」元数据 → 写不可变分片 → 登记 manifest → 合并出会话库 → 一个查询命令 → AI 通过 CLAUDE.md 调用并答对
```

M0 **故意不碰字节码（ASM）**，只读 jar 里两类本就结构化的「白送」文件。M0 要验证的最大风险点是：**分片 + 会话库这条数据通路，以及 Kotlin 抽取器与 Go 查询器之间「以 SQLite 文件为唯一契约」的跨语言边界**。所以即便骨架阶段也用两种语言，各自做到最小。

-----

## 2. 已锁定的技术决策（不要重新评估）

- **抽取器（indexer）**：Kotlin / JVM。M0 版**不依赖 ASM、不依赖 spring-core**，只用 zip 读取 + JSON 解析。
- **查询器（querier）**：**Go**，用 `modernc.org/sqlite`（纯 Go、零 CGO，保证 Windows 与交叉编译无痛）。查询器对 SQLite **只读**。
- **数据格式 / 唯一契约**：SQLite。抽取器写、查询器读，两端只通过 SQLite 文件通信。
- **v1 不含方法级调用图**（M0 更不涉及）。
- **分片身份**：`shard_id = <jar_sha256> + "@" + <extractor_version>`。M0 中 `extractor_version` 常量 = `1`，`schema_version` 常量 = `1`。

## 3. M0 范围

### 3.1 做什么（IN）

1. **抽取器**接受一组 jar（目录 / glob / 逗号分隔列表；可选 fat-jar），对每个 jar：
- 计算 `sha256`；若 manifest 中已有对应分片且文件存在、校验和正确 → **跳过**（增量）。
- 否则抽取下列**两类「白送」元数据**（详见第 8 节）：
  - **自动配置清单**：`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（每行一个 FQN）+ 兼容旧版 `META-INF/spring.factories` 中 `org.springframework.boot.autoconfigure.EnableAutoConfiguration` 这个 key。
  - **配置项元数据**：`META-INF/spring-configuration-metadata.json` 与 `META-INF/additional-spring-configuration-metadata.json`（本就是结构化 JSON）。
- **原子写入**一个分片 SQLite 文件（temp → fsync → rename）。
- 在 manifest 幂等登记（`INSERT OR IGNORE` by `shard_id`）。
1. 抽取器在一次 index 运行结束时，把本次 jar 集合涉及的分片**合并成一个会话库**（session DB），并在项目目录写一个指针文件 `.research4jar/project.json`（含会话库路径、classpath 指纹、覆盖率信息）。
1. **查询器**提供一个命令 `find-config-properties <prefix>`：定位 `.research4jar/project.json` → 打开会话库 → 前缀查询 → 输出 JSON（含 `coverage` 字段）。
1. 抽取器写出 **`CLAUDE.md`** 适配片段（内容见第 12 节），教模型在被问到 Spring 配置项时调用该命令并据 JSON 作答。

### 3.2 明确不做（OUT，不要实现）

- ❌ **任何字节码 / ASM 解析**。不抽取 class / method / field / 注解 / 字符串常量 / Bean 方法 / 条件 / 接口实现。（这些是 M1。）
- ❌ **用 mvn/gradle 自动解析 classpath**。M0 直接喂 jar；构建工具解析推迟到 M3（安装器职责）。
- ❌ **FTS5 全文检索**。`find-config-properties` 用 B-tree 前缀匹配即可。（FTS 是 M1。）
- ❌ **会话库的全局去重 / 按指纹缓存复用 / LRU 淘汰 / GC**。M0「一个项目一个会话库」即可，每次 index 重建无妨。（M2/M5。）
- ❌ **MCP 壳、多宿主探测**。M0 只写 `CLAUDE.md`。
- ❌ **抽取器版本迁移机制 / 分片签名 / 预生成分片下载**。（M3。）
- ❌ 除 `find-config-properties` 外的其他查询命令（`list-auto-configurations` 可作为可选附赠，见 9.2，但不是验收项）。

## 5. 运行时数据布局与路径解析（两端必须一致）

**全局数据目录**解析规则（Kotlin 与 Go 必须算出同一路径）：

1. 若环境变量 `RESEARCH4JAR_HOME` 存在 → 用它。
1. 否则按平台：
- Linux：`$XDG_DATA_HOME/research4jar`，缺省回落 `$HOME/.local/share/research4jar`
- macOS：`$HOME/Library/Application Support/research4jar`
- Windows：`%LOCALAPPDATA%\research4jar`

全局目录内布局：

```
<RESEARCH4JAR_HOME>/
├── manifest.db          # manifest（唯一可写的全局库）
├── shards/              # 内容寻址的分片文件
│   └── <jar_sha256>@<extractor_version>.db
└── sessions/            # 会话库文件
    └── <classpath_fingerprint>.db
```

**项目级**指针文件（由抽取器写入项目根）：

```
<project-dir>/.research4jar/project.json
```

`classpath_fingerprint` 定义：把本次 jar 集合对应的所有 `shard_id` **排序后**用换行连接，取其 `sha256` 的前 16 个十六进制字符。

## 6. 数据流

### 6.1 index（冷路径，可以慢）

```
解析 --jars/--fat-jar 为 jar 文件列表（去重）
for each jar:
    sha256 = hash(jar)
    shard_id = sha256 + "@1"
    if manifest 有该 shard_id 且分片文件存在且校验和匹配:
        skip（计入 skipped）
    else:
        facts = 抽取两类白送元数据
        写分片到 shards/<shard_id>.db（temp → fsync → rename）
        manifest 登记（INSERT OR IGNORE）
计算 classpath_fingerprint
构建会话库 sessions/<fingerprint>.db（见 7.3 / SessionBuilder）
写 <project-dir>/.research4jar/project.json
打印统计：jars_total / indexed / skipped / missing / 耗时
```

### 6.2 query（热路径，必须毫秒级）

```
定位 project.json（--project-dir 指定，或从 cwd 向上查找 .research4jar/project.json）
读出 session_db_path 与 coverage
以只读打开会话库
执行一条带索引的前缀查询
组装 JSON（results + 分页 + coverage）并输出
```

> 注意：查询热路径**绝不重新 hash jar**。会话库与覆盖率都在 index 阶段算好、写进 project.json，查询只是「打开一个 SQLite 文件跑一条 SELECT」。

## 7. 数据模型（M0 子集 DDL）

M0 的表是完整 schema 的**前向兼容子集**：M1 只新增表/列，不改这些。

### 7.1 分片库 schema（每 jar 一个，不可变，只读）

```sql
CREATE TABLE shard_meta (
  jar_coordinate    TEXT,            -- group:artifact:version；私有 jar 可为 NULL
  jar_sha256        TEXT NOT NULL,
  extractor_version INTEGER NOT NULL,
  schema_version    INTEGER NOT NULL,
  created_at        INTEGER,         -- 为可复现，置 0
  class_count       INTEGER          -- M0 写 0 或 NULL（无字节码统计）
);

-- 自动配置 / SPI 注册。M0 只写 mechanism = 'autoconfig.imports'
CREATE TABLE spi_registrations (
  id        INTEGER PRIMARY KEY,
  mechanism TEXT NOT NULL,           -- M0: 'autoconfig.imports'
  key       TEXT,                    -- autoconfig.imports 时为 NULL
  impl_fqn  TEXT NOT NULL            -- 自动配置类的 FQN
);
CREATE INDEX idx_spi_key  ON spi_registrations(key);
CREATE INDEX idx_spi_mech ON spi_registrations(mechanism);

-- 配置项元数据
CREATE TABLE config_properties (
  id          INTEGER PRIMARY KEY,
  prefix      TEXT,                  -- 见 8.2 推导规则
  name        TEXT NOT NULL,         -- 完整配置项名
  type_fqn    TEXT,
  default_val TEXT,                  -- 统一存为字符串；无默认值则 NULL
  description TEXT,
  source_fqn  TEXT                   -- @ConfigurationProperties 类（来自 sourceType）
);
CREATE INDEX idx_cfg_prefix ON config_properties(prefix);
CREATE INDEX idx_cfg_name   ON config_properties(name);
```

> M0 暂不建 `config_properties_fts`（FTS 属 M1）。其余表（classes/methods/fields/annotations/class_interfaces/bean_definitions/conditions/string_constants）M0 不创建。

### 7.2 Manifest schema（全局唯一可写）

```sql
CREATE TABLE shards (
  shard_id          TEXT PRIMARY KEY,   -- jar_sha256@extractor_version
  jar_coordinate    TEXT,
  jar_sha256        TEXT NOT NULL,
  extractor_version INTEGER NOT NULL,
  shard_path        TEXT NOT NULL,      -- 分片文件绝对路径
  shard_checksum    TEXT,               -- 分片文件 sha256
  size_bytes        INTEGER,
  created_at        INTEGER,
  last_access_at    INTEGER,
  source            TEXT                -- M0 恒为 'local'
);
CREATE INDEX idx_shards_sha   ON shards(jar_sha256);
CREATE INDEX idx_shards_coord ON shards(jar_coordinate);
```

> M0 manifest 只含 `shards` 表。`classpath_cache`、`session_cache` 属 M2/M3，不建。
> manifest 连接打开 WAL，并设置 `busy_timeout`（如 5000ms）以容忍并发 index。

### 7.3 会话库 schema（临时，由分片合并而成）

会话库含与分片相同的 `spi_registrations`、`config_properties` 两张表，但每行**额外加一列** `source_shard_id TEXT`，并重建索引：

```sql
CREATE TABLE config_properties (
  id INTEGER PRIMARY KEY, prefix TEXT, name TEXT NOT NULL, type_fqn TEXT,
  default_val TEXT, description TEXT, source_fqn TEXT,
  source_shard_id TEXT NOT NULL
);
CREATE INDEX idx_s_cfg_prefix ON config_properties(prefix);
CREATE INDEX idx_s_cfg_name   ON config_properties(name);

CREATE TABLE spi_registrations (
  id INTEGER PRIMARY KEY, mechanism TEXT NOT NULL, key TEXT, impl_fqn TEXT NOT NULL,
  source_shard_id TEXT NOT NULL
);
CREATE INDEX idx_s_spi_mech ON spi_registrations(mechanism);
```

**合并方式（关键，规避 ATTACH 数量上限）**：逐个分片 `ATTACH → INSERT INTO ... SELECT ..., '<shard_id>' → DETACH`，**任一时刻只挂载 1 个分片**。不要一次性 ATTACH 几百个分片。伪代码：

```
新建空会话库（建上面两张表）
for each shard in 选中分片:
    ATTACH 'shards/<shard_id>.db' AS s
    INSERT INTO config_properties(prefix,name,type_fqn,default_val,description,source_fqn,source_shard_id)
        SELECT prefix,name,type_fqn,default_val,description,source_fqn,'<shard_id>' FROM s.config_properties
    INSERT INTO spi_registrations(mechanism,key,impl_fqn,source_shard_id)
        SELECT mechanism,key,impl_fqn,'<shard_id>' FROM s.spi_registrations
    DETACH s
建索引
```

## 8. 抽取逻辑（两类白送元数据的精确解析）

读 jar 用 `java.util.zip.ZipFile`。**不加载任何类、不执行任何 jar 代码。**

### 8.1 自动配置清单 → `spi_registrations`

- 主来源：zip 条目 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`。逐行读取：`trim` 后跳过空行、跳过以 `#` 开头的注释行；其余每行就是一个自动配置类 FQN。
- 兼容旧来源：zip 条目 `META-INF/spring.factories`（properties 格式）。取 key `org.springframework.boot.autoconfigure.EnableAutoConfiguration` 的值（逗号分隔，可能有以 `\` 续行的多行），拆成 FQN 列表。
- 两来源得到的 FQN **去重合并**后，每个写一行：`mechanism='autoconfig.imports'`、`key=NULL`、`impl_fqn=<FQN>`。

### 8.2 配置项元数据 → `config_properties`

- 来源：zip 条目 `META-INF/spring-configuration-metadata.json` 和 `META-INF/additional-spring-configuration-metadata.json`（两者都可能不存在；存在则都解析）。
- 该 JSON 顶层有 `groups`、`properties`、`hints` 等数组。M0 只用 `properties`（必要时参考 `groups` 推导前缀），忽略 `hints`。
- 对 `properties` 数组每个元素，映射到一行：
  - `name` ← `name`（如 `spring.datasource.url`）
  - `type_fqn` ← `type`（可能含泛型如 `java.util.List<java.lang.String>`，原样存）
  - `description` ← `description`（可空）
  - `default_val` ← `defaultValue`，**统一转字符串**（数字/布尔/字符串都转成其文本形式；数组转 JSON 文本；无则 NULL）
  - `source_fqn` ← `sourceType`（声明该属性的 `@ConfigurationProperties` 类，可空）
  - `prefix` ← 取 `name` 中最后一个 `.` 之前的部分（如 `spring.datasource.url` → `spring.datasource`）；若 `name` 无 `.` 则 `prefix=NULL`。
- 解析容错：单个文件 JSON 解析失败 → 记 warning，跳过该文件，不影响其他文件与该 jar 的分片创建。

### 8.3 坐标（best-effort）

- 尝试读 zip 条目 `META-INF/maven/<group>/<artifact>/pom.properties`，从中取 `groupId`/`artifactId`/`version` 拼成 `group:artifact:version` 作为 `jar_coordinate`。
- 若不存在或不完整 → `jar_coordinate = NULL`；展示时回落用 jar 文件名。
- 若一个 jar 内有多个 `pom.properties`（少见）→ 取第一个即可（M0 不深究）。

### 8.4 「无 Spring 元数据」的 jar 也要建分片

若一个 jar 两类元数据都没有，**仍然创建分片**（只写 `shard_meta`，另两张表为空），并登记 manifest。这样它计入 `jars_indexed`，下次因内容寻址命中而被跳过，不会每次重试。

## 9. CLI 契约

### 9.1 抽取器 `research4jar-index`

```
research4jar-index --jars <DIR|GLOB|逗号分隔列表> [--fat-jar <PATH>] --project-dir <PATH> [--home <DIR>]
```

- `--jars`：可为目录（递归取其中 `*.jar`）、glob、或逗号分隔的多个 jar 路径。可与 `--fat-jar` 并用。
- `--fat-jar`（**M0 可选实现**）：Spring Boot fat jar；解开其中 `BOOT-INF/lib/*.jar` 后逐个当作普通 jar 索引。若时间紧可不实现，但需在 `--help` 标注未实现并清晰报错。
- `--project-dir`：项目根；用于写 `.research4jar/project.json`。
- `--home`：覆盖 `RESEARCH4JAR_HOME`（可选）。
- 行为见 6.1。结束打印统计（建议 JSON 与人类可读各一行，或 `--format` 控制；M0 简单打印即可）。
- 退出码：成功 0；致命错误（如无法写 manifest）非 0。单个 jar 失败不致命。

### 9.2 查询器 `research4jar`

```
research4jar find-config-properties <PREFIX> [--project-dir <PATH>] [--format json|text] [--page N] [--page-size M] [--home <DIR>]
```

- `<PREFIX>`：配置项前缀，如 `spring.datasource`。匹配规则：返回 `config_properties` 中满足 `name = PREFIX` **或** `name LIKE PREFIX || '.%'` 的行（即前缀本身或其下属项），按 `name` 升序。
- `--format`：默认 `json`；`text` 为人类可读表格。
- `--page` / `--page-size`：分页，默认 `1` / `20`。
- 定位 project.json：优先 `--project-dir`；否则从当前工作目录逐级向上查找最近的 `.research4jar/project.json`；找不到 → 输出带清晰 `error` 字段的 JSON 并以非 0 退出。
- 行为见 6.2。

（**可选附赠**，非验收项）`research4jar list-auto-configurations [--format ...] [分页]`：从会话库 `spi_registrations WHERE mechanism='autoconfig.imports'` 列出自动配置 FQN。数据已在库中，实现成本极低，可顺手做。

## 11. 边界情况与健壮性要求

- **损坏 / 非法 zip 的 jar**：捕获异常，跳过，计入 `jars_missing`，记 warning，**不得中断整个 index**。
- **空 jar / 无目标元数据**：见 8.4，仍建分片，计入 `jars_indexed`。
- **重复 jar（同 sha256）**：在输入阶段去重，只处理一次。
- **原子写入**：分片与 manifest 更新都要 temp 文件 → `fsync` → 原子 `rename`，确保半成品永不被读到。
- **并发 index**：manifest 用 `INSERT OR IGNORE`（幂等）+ WAL + `busy_timeout`；分片写入原子。两个进程产出同一分片时互不破坏。
- **会话库可重建**：M0 每次 index 重建会话库即可（覆盖同名文件用原子 rename）。
- **查询热路径**：只读打开会话库；不重新 hash jar；不写任何东西（`last_access_at` 更新等留到 M2）。
- **`jars_total` / `jars_indexed` / `jars_missing` 定义**：`total` = 去重后输入 jar 数；`indexed` = 成功打开并处理（含「无元数据」）的 jar 数；`missing` = 因损坏/不可读而失败的 jar。三者满足 `indexed + len(missing) = total`。

## 14. 验收标准（Definition of Done）

M0 完成的判定（建议据此写集成测试）：

1. **端到端跑通**：给定一个真实 Spring Boot 应用的依赖 jar 目录（可由 `mvn dependency:copy-dependencies` 预先收集），`research4jar-index --jars <dir> --project-dir <proj>` 无异常完成，生成 `shards/*`、`manifest.db`、`sessions/*`、`<proj>/.research4jar/project.json`，并打印含 total/indexed/skipped/missing 的统计。
2. **查询正确**：`research4jar find-config-properties spring.datasource` 返回正确的配置项 JSON（含 `spring.datasource.url` 等），且响应含 `coverage`。
3. **增量命中**：紧接着第二次跑同样的 index，所有分片因内容寻址命中而被 **skipped**，耗时显著低于首次。
4. **AI 可用**：在 Claude Code 中打开该项目并问「`spring.datasource.hikari.maximum-pool-size` 是什么、默认值多少」，模型据 `CLAUDE.md` 指引运行命令并答对（数值取自命令输出而非臆测）。
5. **健壮**：输入目录里放入一个损坏 jar（如把任意文本文件改名为 `.jar`），index 不崩溃，该 jar 出现在 `jars_missing` 中，其他 jar 正常索引。
6. **跨语言契约成立**：抽取器（Kotlin）写出的分片/会话库，查询器（Go）能正确读取——即 SQLite 文件契约在两种语言间无缝工作。
7. **确定性**：对同一个 jar 抽两次，分片内容（除文件系统元信息外）一致。
