# M6: Pure-Java Consolidation

**Status:** implementation complete; first Maven Central publication pending
**Decision date:** 2026-07-04
**Supersedes:** the Go querier (`querier/`) and the goreleaser binary distribution.

## Why

Research4Jar serves Java engineers. M6 rebuilds the product around two values:

1. **开箱即用，极度简易** — zero-install onboarding. A Java engineer must be able to adopt the tool through the toolchain they already run (Maven Central, `mvn`/Gradle plugins), with no curl, no PATH edits, no separate binary download. This also unblocks enterprises whose networks allow Nexus/Central mirrors but not GitHub binary downloads.
2. **好用，极致性能** — the fast paths agents actually hit must be fast: warm MCP calls ≈1 ms, typical installed-launcher warm CLI calls around 100–130 ms on the recorded final setup, indexing seconds not minutes, and no gigabyte-scale session files.

Measured facts driving the decision (2026-07-04 benchmark, 99 real Spring Boot 3.5.13 jars / 155 MB, Apple Silicon, session 1.0 GB):

| Path | Go (modernc SQLite) | JVM (sqlite-jdbc = C SQLite) |
| --- | --- | --- |
| Warm point query | 1.8 ms (MCP) | 3.7 ms first, <1 ms after |
| 1.1M-row `LIKE '%…%'` scan | 960 ms | 130–139 ms (**7× faster**) |
| CLI one-shot | 28 ms | ~700 ms naive / **50–60 ms via daemon thin client** (measured floor) |

The JVM stack is faster everywhere that matters except cold CLI start, which the daemon (Phase 2.5)
solves. Consolidating on one Kotlin codebase also deletes the Kotlin/Go dual-writer session
contract and its drift-guard e2e machinery.

## Target architecture

```
core/            dev.research4jar:research4jar-core
                 extraction (ASM), shards, manifest, session build (single writer),
                 query engine, registry client/export/seed, cache lifecycle,
                 classpath discovery, dependency provenance, doctor
cli/             dev.research4jar:research4jar-cli  (shadow fat jar)
                 arg parsing/dispatch, MCP stdio server, daemon + thin client,
                 self-install (~/.research4jar/bin + lib)
maven-plugin/    dev.research4jar:research4jar-maven-plugin
                 goals: index, doctor. Classpath from project.getArtifacts(),
                 provenance from Maven's DependencyGraphBuilder — no subprocess.
gradle-plugin/   dev.research4jar.gradle (marker) / research4jar-gradle-plugin
                 task research4jarIndex; runtimeClasspath + native dependency
                 graph → Gradle provenance ships here (closes the M-gap).
```

- **Bytecode target: Java 8** for `core`/`cli`/plugins (tests stay on 17 for Spring Boot 3
  fixtures). "Any JVM that can run your project's Maven can run us."
- **SQLite:** xerial sqlite-jdbc everywhere (C SQLite; native libs bundled per platform).
- **JSON:** Jackson (already a core dependency).
- **Ed25519 (registry signatures):** `net.i2p.crypto:eddsa` (pure Java 8, ~100 KB) — JDK EdDSA
  needs Java 15+.
- **No Go, no CGO, no goreleaser.** One language, one build (Gradle), one artifact family.

### Distribution

Primary: **Maven Central**. Zero-install entry points:

```bash
mvn dev.research4jar:research4jar-maven-plugin:index      # inside any Maven project
# Gradle: plugins { id("dev.research4jar") } → ./gradlew research4jarIndex
```

Self-install (`mvn …:index` copying the fat jar to `~/.research4jar/lib/` and writing
`~/.research4jar/bin/research4jar` (+ `.cmd`) launchers plus printing the
`claude mcp add` line) **lands with the first Central release** — before the CLI artifact
exists on Central the plugin has nothing to install. Until then the Maven plugin prints the
MCP setup hint and the CLI comes from a GitHub Release or `install.sh`.

Secondary: the fat jar attached to GitHub Releases (`java -jar research4jar-cli.jar …`).

### Daemon (CLI latency) — as implemented

- `research4jar daemon`: same fat jar; listens on `127.0.0.1:<ephemeral>`; auth via
  one atomically published `<data-home>/daemon/<protocol>.endpoint` file (0600,
  containing generation, port, token, and build identity); idle-exit after 30 min.
  Bounded connection workers authenticate and acknowledge independently, while only
  `runCli` execution is serialized. A slow query therefore does not block another
  client's handshake (no session-handle cache: C-SQLite opens are cheap enough that
  measured latency did not justify one — the spec'd per-project LRU was deliberately
  dropped).
- Thin client = the fat jar's entry class touching only JDK classes (jackson and
  sqlite-jdbc never load on the fast path). Daemon alive → mutually authenticate, then
  forward `{build, cwd, argv}` under a request MAC;
  daemon absent → run in-process (~1 s once) and spawn the daemon in the background.
  Query commands only; `RESEARCH4JAR_NO_DAEMON=1` opts out. A custom
  `RESEARCH4JAR_HOME` naturally selects a separate endpoint directory and child
  environment, so custom homes retain the warm path without crossing streams.
  `status --check-classpath` bypasses the daemon because its build-tool probe
  must inherit the current invocation's PATH/JAVA_HOME/build-tool variables.
- Protocol (revised from the original JSON-frame sketch for client weight): `R4JD3`
  performs a nonce-based HMAC challenge/response. The server proves possession of the
  endpoint token before the client sends its cwd or argv, then the client authenticates
  the complete build/cwd/argv request; the token itself never crosses the socket.
  Successful requests receive `A <build-id>` followed by `O <len>\n<bytes>` /
  `E <len>\n<bytes>` / `X <code>` frames. Authenticated `M BUILD`, `M PROTOCOL`, and
  `M COMMAND` handshakes reject obsolete builds, incompatible peers, and commands outside
  the read-only daemon allowlist. Protocol-keyed endpoint discovery lets a newly built
  client authenticate and retire an older daemon immediately. Atomic rename plus a
  cross-process lock makes publish/conditional cleanup generation-safe; a separately
  locked, expiring start lease prevents concurrent cold clients from spawning duplicates,
  including delayed-starter ABA races. Total request/write deadlines and request-line,
  frame, per-response, and aggregate response limits bound slow or oversized peers.
  Response bytes remain buffered until a valid terminal `X`, so a broken daemon cannot
  leak partial output before the cold fallback runs. Server-side stdout/stderr share the
  same bounded capture budget, preventing an oversized handler result from exhausting
  the daemon before the client can fall back. A lone connection refusal is also re-probed
  with short backoff while the endpoint generation is unchanged, avoiding false stale-file
  cleanup during accept-backlog or file-descriptor bursts.
- Requests with `--page-size` above 100 deliberately stay on the ordinary cold path.
  The public CLI limit remains 1000 and output is unchanged; this keeps the shared
  256 MiB warm host from retaining the largest legal response models.
- Final full-v8 measurements put typical installed-launcher warm queries at **100–130 ms**,
  with byte-identical output vs the cold path. Earlier, smaller-session runs measured
  90–116 ms. These are observed ranges, not a claim that final p95 latency is below 100 ms.

## Compatibility contract (the port gate)

- During the port, command names, flags, JSON **key sets and values**, exit codes, and English
  stderr messages were matched against the former Go querier. `tests/e2e.sh` remains the
  structural contract suite and now drives the Kotlin/JVM CLI through both index and query
  shims; former dual-writer checks are route-parity checks for local-extraction and
  registry-backed indexing.
- Shards: unchanged (`extractor_version` stays 2). Registry layout unchanged.
- Session: layout version 8 adds shard/range bookkeeping, bounded delta depth,
  class/method/string trigram shadows, the method-owner string index, and compact
  ordered INTEGER keys for repeated source-shard references. Repeated method
  `owner#name` values are derived at query time; a compact resolution bit keeps
  malformed-orphan NULL semantics. The canonical shard
  hash remains in `session_shards` and is restored before JSON attribution. Older
  sessions rebuild on the next index automatically. The exact schema/view/index
  contract is validated before reuse. `CLAUDE.md` snippet and `project.json` are unchanged.
- Session readers and delta copies hold a fair cross-process shared lease; GC
  takes the exclusive side and rechecks mtime before unlinking. This keeps a
  concurrently used content-addressed path valid on Unix and Windows. A fixed
  directory-level lifecycle coordinator makes deleted sessions' per-session lock
  sidecars reclaimable without inode-ABA races, and local lease states disappear
  after their final holder closes.
- Pagination accepts at most 1000 rows per page inside a 100000-result window;
  all SQL offsets use checked `Long` arithmetic. CLI and MCP enforce the same contract.

## Performance work (language-independent, lands first)

| Fix | Baseline → target |
| --- | --- |
| Default `-Xmx512m` for the indexer | peak RSS 2.09 GB → ~550 MB (measured: no time penalty) |
| `search_symbols` de-duplication (view or thin table) | session 1.07 GB → ≤ 500 MB; build −40–60% |
| Set-based derived columns (was ~620k row UPDATEs) | part of the above build win |
| Derive method `owner#name`; remove repeated column/index | −110.72 MiB on the 687696-method session; exact lookup level |
| search-symbol: strict score tiers, adaptive trigram/scan fallback | 960 ms → <20 ms in-process typical; installed warm CLI ~100–130 ms, no-hit ~340–370 ms, while dense posting lists automatically retain the faster scan |
| Skip `mvn dependency:tree` when classpath fingerprint unchanged | warm real-project re-index loses a full Maven run |
| Cold index 99 jars | 12.3 s → ≤ 8 s on the JVM path, ≤ 6 s on the registry path (session fixes; extraction already parallel) |

**Phase 0 measured outcome (2026-07-04):** session 387 MB, search-symbol 30 ms end-to-end
(≈4 ms query), cold index 7.8 s, registry-path index 5.2 s (was 13.0 — now faster than JVM
extraction, as designed), peak RSS 812 MB with the default `-Xmx512m` cap, MCP unchanged at
1.8 ms, e2e green. Ranking change shipped with it: prefix tiers (class 82/80, annotation 78,
spi 76, config 70, string 60) now rank above the contains tier and match case-sensitively via
index ranges; case-insensitive prefix hits still surface in the contains tier.

**Follow-up measured outcome (2026-07-14):** adaptive class/method/string FTS keeps dense
postings on the faster scan, and `search-symbol` consumes mutually exclusive score tiers until
the page is full. The method shadow stores `name,descriptor` rather than repeated
`owner#name,descriptor`; method symbols are then projected from owner plus name, and repeated
source-shard hashes use ordered integer keys. The final real 222-jar session-only build is
6.72 s / 297.56 MiB (from 10.37 s / 528.3 MiB) with 222.7 MiB peak RSS. Installed warm
typical queries remain around 100–130 ms, no-hit search is about 340–370 ms, and the
100000-result-window boundary page is about 0.40–0.43 s. A vacuumed A/B attributes 110.72 MiB
of the reduction to the removed method-symbol column/index; malformed rows retain only a
0.34 MiB owner-resolution marker.

Historical implementation note: M6 evaluated a flattened UNION-ALL view against a thin-table
fallback. The shipped v8 keeps the view and avoids whole-view materialization through mutually
exclusive score tiers and hits-driven indexed slices; the final measured outcomes are recorded
above.

## Historical phases and acceptance

- **Phase 0 — perf + baseline (at that point the Go stack was still primary).**
  The table above, plus Java 8 retarget of the indexer. Gate: `make test`, `make e2e`,
  benchmark re-run hits the numbers.
- **Phase 1 — the port.** 1a modules (`core`/`cli`, shadow jar) → 1b query engine → 1c MCP →
  1d registry+cache → 1e classpath/depgraph/doctor/CLI. Gate (1f): `tests/e2e.sh` fully green
  with `RESEARCH4JAR_QUERY` = the Java CLI; unit tests for splitFQN SQL, pagination, TGF parse.
- **Phase 2 — build plugins.** Gradle plugin (TestKit-tested; native provenance = Gradle
  provenance milestone) and Maven plugin (in-process classpath + graph; smoke test via fixture
  `mvnw` where network allows, else CI).
- **Phase 2.5 — daemon.** Original gate: the e2e query suite passes in daemon mode and a warm
  CLI probe is below 100 ms. The final full-v8 recorded outcome is a typical 100–130 ms range;
  this document does not claim a final sub-100-ms percentile.
- **Phase 3 — cutover.** maven-publish + signing config (publishing itself needs Sonatype
  credentials — user action), README/CHANGELOG rewrite, CI/release workflows to Gradle, remove
  `querier/` + `.goreleaser.yaml` (only permitted if 1f is fully green), `install.sh` builds the
  Java stack, registry-publish workflow uses `java -jar … registry seed`.

Historical rollout rule: the Go stack was deleted only at Phase 3 after the parity gate;
everything before that point was additive. Git history retains the querier at `pre-m6-go`.

## Risks

| Risk | Mitigation |
| --- | --- |
| Port defects in query semantics | e2e structural-JSON gate + verbatim SQL port + targeted unit tests |
| SQLite view planner materializes | thin-table fallback, acceptance by numbers |
| Daemon lifecycle bugs (zombies, stale ports, races) | Protocol-keyed, generation-tagged endpoint; nonce/HMAC mutual authentication; cross-process locks; idle suicide; cold-run fallback |
| Jackson class-load cost on CLI | only the daemon/thin-client path is latency-critical; heavy classes load in the daemon |
| Ed25519 on Java 8 | net.i2p.crypto:eddsa (verify+sign), covered by existing e2e registry section |
| Maven Central publishing friction | config ships in Phase 3; actual publish is a documented user action (namespace + GPG) |
| Windows | TCP-loopback daemon (no unix sockets), `.cmd` launchers, e2e stays POSIX (CI matrix later) |
