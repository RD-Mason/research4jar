# M6: Pure-Java Consolidation

**Status:** in progress
**Decision date:** 2026-07-04
**Supersedes:** the Go querier (`querier/`) and the goreleaser binary distribution.

## Why

Research4Jar serves Java engineers. M6 rebuilds the product around two values:

1. **开箱即用，极度简易** — zero-install onboarding. A Java engineer must be able to adopt the tool through the toolchain they already run (Maven Central, `mvn`/Gradle plugins), with no curl, no PATH edits, no separate binary download. This also unblocks enterprises whose networks allow Nexus/Central mirrors but not GitHub binary downloads.
2. **好用，极致性能** — the fast paths agents actually hit must be fast: warm MCP calls ≈1 ms, warm CLI one-shots <100 ms, indexing seconds not minutes, and no gigabyte-scale session files.

Measured facts driving the decision (2026-07-04 benchmark, 99 real Spring Boot 3.5.13 jars / 155 MB, Apple Silicon, session 1.0 GB):

| Path | Go (modernc SQLite) | JVM (sqlite-jdbc = C SQLite) |
| --- | --- | --- |
| Warm point query | 1.8 ms (MCP) | 3.7 ms first, <1 ms after |
| 1.1M-row `LIKE '%…%'` scan | 960 ms | 130–139 ms (**7× faster**) |
| CLI one-shot | 28 ms | ~700 ms naive / **50–60 ms via daemon thin client** (measured floor) |

The JVM stack is faster everywhere that matters except cold CLI start, which the daemon (Phase 2.5)
solves. Consolidating on one Kotlin codebase also deletes the Kotlin/Go dual-writer session
contract and its drift-guard e2e machinery.

## Target architecture

```
core/            dev.research4jar:research4jar-core
                 extraction (ASM), shards, manifest, session build (single writer),
                 query engine, registry client/export/seed, cache lifecycle,
                 classpath discovery, dependency provenance, doctor
cli/             dev.research4jar:research4jar-cli  (shadow fat jar)
                 arg parsing/dispatch, MCP stdio server, daemon + thin client,
                 self-install (~/.research4jar/bin + lib)
maven-plugin/    dev.research4jar:research4jar-maven-plugin
                 goals: index, doctor. Classpath from project.getArtifacts(),
                 provenance from Maven's DependencyGraphBuilder — no subprocess.
gradle-plugin/   dev.research4jar.gradle (marker) / research4jar-gradle-plugin
                 task research4jarIndex; runtimeClasspath + native dependency
                 graph → Gradle provenance ships here (closes the M-gap).
```

- **Bytecode target: Java 8** for `core`/`cli`/plugins (tests stay on 17 for Spring Boot 3
  fixtures). "Any JVM that can run your project's Maven can run us."
- **SQLite:** xerial sqlite-jdbc everywhere (C SQLite; native libs bundled per platform).
- **JSON:** Jackson (already a core dependency).
- **Ed25519 (registry signatures):** `net.i2p.crypto:eddsa` (pure Java 8, ~100 KB) — JDK EdDSA
  needs Java 15+.
- **No Go, no CGO, no goreleaser.** One language, one build (Gradle), one artifact family.

### Distribution

Primary: **Maven Central**. Zero-install entry points:

```bash
mvn dev.research4jar:research4jar-maven-plugin:index      # inside any Maven project
# Gradle: plugins { id("dev.research4jar") } → ./gradlew research4jarIndex
```

First `index` self-installs the CLI: copies the fat jar to `~/.research4jar/lib/`, writes
`~/.research4jar/bin/research4jar` (+ `.cmd`) launchers, and prints the MCP registration line:

```bash
claude mcp add research4jar -- ~/.research4jar/bin/research4jar mcp
```

Secondary: the fat jar attached to GitHub Releases (`java -jar research4jar-cli.jar …`).

### Daemon (CLI latency)

- `research4jar daemon`: same fat jar; listens on `127.0.0.1:<ephemeral>`; auth via
  `~/.research4jar/daemon/<version>.token` (0600) + `<version>.port`; idle-exit after 30 min;
  per-project session handle LRU (cap 4) with `project.json` re-read per request (pointer is the
  correctness boundary; session files are content-addressed and immutable).
- Thin client = the fat jar's entry class importing only sockets (measured 50–60 ms JVM
  round-trip). Daemon alive → forward `{token, cwd, argv}`, stream stdout/stderr frames, exit
  with the daemon-reported code. Daemon absent → run in-process (~0.7 s once) and spawn the
  daemon in the background for next time. `RESEARCH4JAR_NO_DAEMON=1` opts out.
- Protocol: newline-delimited JSON frames `{"fd":1,"data":…}` / `{"exit":0}`. Version-keyed
  socket files; mismatched daemons are ignored and idle out.

## Compatibility contract (the port gate)

- Command names, flags, JSON **key sets and values**, exit codes, and English stderr messages
  match the Go querier. `tests/e2e.sh` asserts structural JSON equality and drives whichever
  binaries `RESEARCH4JAR_INDEX`/`RESEARCH4JAR_QUERY` point at — it is the parity gate and must
  pass against the Java CLI without test edits (except where dual-writer assertions become
  single-writer tautologies).
- Shards: unchanged (`extractor_version` stays 2). Registry layout unchanged.
- Session: schema changes below bump the session layout version once; sessions rebuild on next
  index automatically. `CLAUDE.md` snippet and `project.json` unchanged.

## Performance work (language-independent, lands first)

| Fix | Baseline → target |
| --- | --- |
| Default `-Xmx512m` for the indexer | peak RSS 2.09 GB → ~550 MB (measured: no time penalty) |
| `search_symbols` de-duplication (view or thin table) | session 1.07 GB → ≤ 500 MB; build −40–60% |
| Set-based derived columns (was ~620k row UPDATEs) | part of the above build win |
| search-symbol: indexed branches first, contains-scan fallback | 960 ms → <20 ms typical |
| Skip `mvn dependency:tree` when classpath fingerprint unchanged | warm real-project re-index loses a full Maven run |
| Cold index 99 jars | 12.3 s → ≤ 6 s (session fixes; extraction already parallel) |

`search_symbols` note: try the UNION-ALL view first; SQLite must flatten it and push predicates
into per-branch base-table indexes (verify with `EXPLAIN QUERY PLAN`). If the planner
materializes, fall back to a thin table (name/simple/package/score + ref ids only, join detail at
page time). Acceptance is by number, not mechanism: session ≤ 500 MB and typical search < 20 ms
with no regression in find-class/list-packages.

## Phases and acceptance

- **Phase 0 — perf + baseline (this tree, Go stack still primary).**
  The table above, plus Java 8 retarget of the indexer. Gate: `make test`, `make e2e`,
  benchmark re-run hits the numbers.
- **Phase 1 — the port.** 1a modules (`core`/`cli`, shadow jar) → 1b query engine → 1c MCP →
  1d registry+cache → 1e classpath/depgraph/doctor/CLI. Gate (1f): `tests/e2e.sh` fully green
  with `RESEARCH4JAR_QUERY` = the Java CLI; unit tests for splitFQN SQL, pagination, TGF parse.
- **Phase 2 — build plugins.** Gradle plugin (TestKit-tested; native provenance = Gradle
  provenance milestone) and Maven plugin (in-process classpath + graph; smoke test via fixture
  `mvnw` where network allows, else CI).
- **Phase 2.5 — daemon.** Gate: e2e query suite passes in daemon mode; warm CLI probe < 100 ms.
- **Phase 3 — cutover.** maven-publish + signing config (publishing itself needs Sonatype
  credentials — user action), README/CHANGELOG rewrite, CI/release workflows to Gradle, remove
  `querier/` + `.goreleaser.yaml` (only permitted if 1f is fully green), `install.sh` builds the
  Java stack, registry-publish workflow uses `java -jar … registry seed`.

Rollback: the Go stack is deleted only at Phase 3 after the parity gate; everything before is
additive. Git history retains the querier; tag `pre-m6-go` before removal.

## Risks

| Risk | Mitigation |
| --- | --- |
| Port defects in query semantics | e2e structural-JSON gate + verbatim SQL port + targeted unit tests |
| SQLite view planner materializes | thin-table fallback, acceptance by numbers |
| Daemon lifecycle bugs (zombies, stale ports, races) | Gradle/mvnd-proven patterns: version-keyed files, token auth, idle suicide, cold-run fallback always works |
| Jackson class-load cost on CLI | only the daemon/thin-client path is latency-critical; heavy classes load in the daemon |
| Ed25519 on Java 8 | net.i2p.crypto:eddsa (verify+sign), covered by existing e2e registry section |
| Maven Central publishing friction | config ships in Phase 3; actual publish is a documented user action (namespace + GPG) |
| Windows | TCP-loopback daemon (no unix sockets), `.cmd` launchers, e2e stays POSIX (CI matrix later) |

## User-visible outcome

| | Before (M5) | After (M6) |
| --- | --- | --- |
| Install | download archive, PATH, doctor | one `mvn`/Gradle line (or `java -jar`) |
| Runtime needs | Go binary + Java 11+ for indexing | any Java 8+ JVM (the one running Maven suffices) |
| CLI warm query | 28 ms | 50–100 ms (daemon) |
| MCP warm query | 1.8 ms | ≈1 ms |
| Worst-case search | 960 ms | <20 ms typical / ~130 ms scan floor |
| Session on disk | 1.07 GB | ≤ 500 MB |
| Index peak RSS | 2.09 GB | ~550 MB |
| Cold index (99 jars) | 12.3 s | ≤ 6 s |
| Gradle dependency provenance | unsupported | native via Gradle plugin |
